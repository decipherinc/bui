Buttons

Button Types

Use as described. Remember that links, divs and spans are not buttons.

Standard

Usage: Used as a generic button. Should have simple text that describes the action.

Rules: May include an appropriate icon with text.

You can add btn-darken to standard buttons when extra contrast is needed.

<button class="btn btn-default">Standard </button>

<button class="btn btn-default" disabled="disabled">Disabled </button>

<button class="btn btn-default icon-add"> Icon + Text</button>

<button class="btn btn-default"><i class="icon-add"></i>Text w/ Icon in 'i' tag</button>

<button class="btn btn-default btn-darken">Darker Standard Button</button>

Note: Both an <i> tag or a <span> tag can be used for icon.

Primary

Usage: Used when there’s a need for a suggested action, like the “Done” button in a modal or an obvious "next step".

In situations where the primary button is for an undoable action (this normally occurs in a confirmation modal or tooltip), you can add the class dangerous to make the button red.

Rules: There shouldn’t be more than one primary button in any dialog/modal/interface.

<button class="btn btn-primary">Primary </button>

<button class="btn btn-primary" disabled="disabled">Disabled </button>

<button class="btn btn-primary dangerous"> Primary</button>

<button class="btn btn-primary dangerous" disabled="disabled"> Disabled</button>

Special Action

Usage: Used in an application to denote a special action. It has a “go” connotation, so it should be used to initiate a journey of some sort (like “new project”, “new campaign”, “launch campaign”).

Rules: Maximum of one per application (although having none is fine).

<button class="btn btn-special">Special </button>

<button class="btn btn-special" disabled="disabled">Disabled </button>

Icon Buttons

Usage: Only for commonly used and easy to understand actions in the main area of an application, otherwise use a standard or “icon + label” button with descriptive text.

Special note on "Edit" icon button: The edit icon () may use btn-text instead, making it a blue icon button, but only when the edit button sends the user away for editing (eg. opens a modal for editing).

Rules: Icon buttons must have descriptive tooltips that appear instantly on hover.

Text Button

Usage: Used to indicate a supporting action, typically inside of a tab or accordion menu.

Rules: To avoid being confused with a standard link, these buttons must always be accompanied with an appropriate icon.

<button class="btn btn-text icon-info">Icon + Label</button>

<button class="btn btn-text"><span class="icon-info">Label with icon in span</span></button>

<button class="btn btn-text icon-info" disabled="disabled"> Disabled</button>

<button class="btn btn-text btn-text-flush icon-info">Padding Flushed</button>

Note: Both an <i> tag or a <span> tag can be used for icon.

Expand Toggle Button

Usage: Used for toggeling content in or out of the viewport.

Rules: The caret icon always point towards the text when collapsed, when content is opened, the arrow points towards the content.

Some fancy content.

<button class="btn btn-text-toggle toggle-right"
  id="example-toggle-button">Click to show content below</button>
<div id="example-content">Some fancy content.</div>

<script>
 $('#example-toggle-button').on('click', function() {
  $(this).toggleClass('down').toggleClass('up');
  $('#example-content').slideToggle('fast');
 });
</script>

Code Note: To animate and change the direction of the arrow box, just add/change it's directional class (arrow points down by default or with now direction declared). .toggle-up, .toggle-right, .toggle-down, and .toggle-left are all available. So in the example above, the button has the .toggle-down class added by default, then when the user clicks, the .toggle-down class is removed and the .toggle-up class is added (sweet css animations built in!).

Toggle Buttons

Usage: Used like a checkbox to filter viewport information.

<button class="btn btn-toggle" data-toggle="button">Toggle Button</button>

<button class="btn btn-toggle" disabled="disabled">Toggle Button</button>

Links

Usage: Links send the user away from the current page to somewhere else.

Rules:

  • Links should not look like buttons, they always display an underline on hover.
  • When a link’s destination is to another Beacon application, it may be accompanied with the icon-go-to for internal links (), otherwise a link should not have any associated icons.
  • The <a href=”#”> markup is prefered for links, but if for some reason you need to, you can style buttons as links using <btn-link> class. Just note that links with button styling have button padding (so they align nicely to buttons).
A Standard Link

<button class="btn btn-link">Button Link</button>

<a href="#URL">A Standard Link </a>

Cancel

Usage: Used in any dialog box/modal/menu where actions require a commit button, there should always be an option to cancel as well. Cancel should back out any changes that occurred and leave it in the previous state.

Rules: Should always be placed to the left of a Primary of standard button.

<button class="btn btn-cancel"> Cancel</button>

<button class="btn btn-cancel" disabled="disabled">Cancel </button>

Delete or Remove

Usage: A stand alone “x” icon (never the letter "x" or "X") used to remove or delete an item.

Rules: Typically placed on the far right side in table rows. It may be appropriate to have a confirmation alert if the action cannot be undone. While the hover state is always @color-red, the default state may need to be a different color depending on what it's containing background color is.

<button class="btn btn-remove"><i class="icon-remove"></i></button>

<button class="btn btn-remove" disabled="disabled"><i class="icon-remove"></i></button>

Tags

Standard Tags

Usage: Tags are used for representing data attributes on an object of some sort. In the Portal, they're used for classifying and organizing projects, while in other areas of Beacon they may be used for displaying applied filters.

Rules: Tags are modular and must always be removable by hovering over the tag and clicking the remove button.

Decipher Inc
<div class="tag">Decipher Inc<button class="btn btn-remove"><i class="icon-remove"></i></button></div>

Code Note: You can turn a <div>, <li> or other container into a tag, but a btn-remove needs to exist inside after the tag's name for the remove button to work.

Group Tags

Usage: Same as a standard tag, only a group tag is meant to represent a pre-defined group of users that can be added to a project all at once using the tag system. They are styled slightly differently so users can distinguish them standard organizational tags.

BUI Group
<div class="tag icon-user-group"> BUI Group<button class="btn btn-remove"><i class="icon-remove"></i></button></div>

Code Note: All you have to do to turn a tag into group tag is add the icon-user-group class. This adds the icon and comes packaged with some minor css stuff.

Header

Title + Optional Drop-Down Menu:

Usage: The title should be short and descriptive. This title may be displayed as a Drop-Down menu in situations where the modal features a number of categories (e.g. the project menu in the survey building app).

The title may also contain a left aligned icon before the text in situations like a warning dialog.

Help Resources

Usage: There may be a “Learn More” link or help icon right aligned in the header.

Close "x" Button

Usage: A stand alone icon-close (never the letter "x" or "X") used to close out of a modal, should always appear in the top right corner.

Rules: Equivalent to cancel, any changes made and not executed should be reverted to the previous state.

Content Area

Usage: Any associated content should be displayed in here.

Rules: Do not use panel or accordion menus inside modals.

Footer

Usage: The footer of a modal should contain any buttons related to submitting, canceling, or saving the content. There may also be checkboxes that appear for settings related to the modal (e.g. “Show this screen for new projects”).

Rules: The primary button should always be the furthest right aligned, with secondary buttons to the left of that, and a cancel button to the left of all buttons. Any checkboxes in the footer should be aligned to the left.

Placement

The standard modal is placed 30px from the top and will allow scrolling if the modals content is long (or if the user is on a device with a small viewport).

If your modal has very little content and you're sure the user won't need the ability to scroll, you can apply the custom BUI modal-centered class to .modal, and center the modal in the window. Don't use this for any modal containing complex menues of any sort. The centered modal uses the transform: translate hack and can cause alignment anomalies. Best to only use this for modals displaying dialog text.

If you need your modal to scroll the body of the modal instead of the viewport, add the class .modal-scroll-body to .modal.

Fly Out Modal

Usage: This modal is used for things like adding filters. This is a good option when a modal is appropriate and allowing the user to continue scrolling the main stage with the modal open is useful.

Rules: When the fly-out-modal is visible, your javascript will need to hide it when the user clicks away from the modal, like the example below and this web page's main menu. Also when the user takes an action in the modal (e.g. clicks the 'x' close button, primary button.

Use javascript to toggle the class .fly-out-opened on .fly-out-modal to show or hide the modal.

Add the class .fly-out-scroll to .fly-out-menu, if you know scrolling will take place, to add top/bottom shadows on the body of the modal.

Add the class .flush to .fly-out-body to set it's padding to 0.

Typography

Most of this uses the same markup as bootstrap, but it's worth reading through as it also provides some best practice tips.

Font

BUI uses em units heavily to maintain proportionality of design elements to the text. Applications in Beacon have been built on inconsistent root font sizes, so nothing in BUI relies on rem units and should scale appropriately to the previously set body font size.

When adding BUI typography, the root <html> font-size is set to 87.5% (14px on standard 16px based browsers), and the <body> font-size to 14px. If the application you're working on needs to override these default BUI font-size settings, you'll need to match the level of BUI specificity in an override like so:

.bui-typography {
  font-size: 87.5%; /* Change to your desired root font-size */
}
.bui-typography body {
  font-size: 14px; /* Change to your desired body font-size */
}

The font family used in BUI is Open Sans.

The font weights available are 400 (+ italic), 600 (semi-bold), and 700 (bold).

Some designs may call for an unbolded heading. Add the class unbold to the <h*> tag.

H2 Heading

H3 Heading

etc . . .

<h1 class="unbold">H1 Heading</h1>

<h2 class="unbold">H2 Heading</h2>

<h3 class="unbold">H3 Heading</h3>

All headers and titles should be written using title casing. As there are a few different ways of applying title casing, you can use this handy tool to make sure you’ve got it right. The rules we use for title casing come from the Chicago Manual of Style.

Emphasis

You can use HTML's default emphasis tags.

Small

Use the <small> tag for de-emphasizing text.

This might be useful for fine print text.

<small> . . . </small>

Bold

Use the <strong> tag for bold emphasis.

Making text bold is great for emphasis. Just remember, if too much text is bold, no text is bold.

<strong> . . . </strong>

Italic

Use the <em> tag for italic emphasis.

Italic text is good for emphasising a single word, or a whole phrase.

<em> . . . </em>

All Caps

Add class="caps" for all caps.

All caps may be good for certain section titles - they're loud, so use sparingly

<h4 class="caps"> . . . </h4>

Underline

Never use underline. Links underline themselves on hover, that's it.
